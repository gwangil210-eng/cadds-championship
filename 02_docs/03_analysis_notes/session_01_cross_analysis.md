### 2.1 3×3 교차점 식별

### 2.2 교차점 상세 분석

#### C1: Agent Hooks ↔ CADSS 정책 엔진 (구조적 동형)

이것이 본 연구의 핵심 발견이다.

**구조 대응표:**

| Agent Hooks 이벤트 | CADSS 정책 엔진 상태 | 공통 패턴 |
|-----------------|-----------------|---------|
| SessionStart | 요청 수신 + 인증/권한 | 컨텍스트 초기화 |
| PreToolUse | 불확실성 평가 + 정책 판정 | 실행 전 게이팅 |
| PermissionRequest(allow) | AUTO | 자동 승인 |
| PermissionRequest(review) | REVIEW | 인간 검토 요청 |
| PermissionRequest(escalate) | APPROVAL | 상위 권한 승인 |
| PermissionRequest(deny) | BLOCK | 실행 차단 |
| PostToolUse | 감사 로그 기록 | 실행 후 기록 |
| Stop | 세션 마감 + 로그 체인 마감 | 정리 작업 |

**동형 증명 (비형식적):**

두 시스템을 상태 머신으로 모델링하면:

```
Agent Hooks 상태 공간: S_h = {Init, PreExec, Permission, PostExec, Done}
CADSS 정책 공간: S_c = {Received, Evaluated, Decided, Executed, Logged}

사상 함수 φ: S_h → S_c
  φ(Init) = Received
  φ(PreExec) = Evaluated
  φ(Permission) = Decided
  φ(PostExec) = Executed
  φ(Done) = Logged

전이 함수 보존:
  Agent에서 PreExec → Permission(deny) 이면
  CADSS에서 Evaluated → Decided(BLOCK) 이다.
  
  즉, φ(δ_h(s, a)) = δ_c(φ(s), a) 가 모든 상태-액션 쌍에서 성립한다.
```

**연구 공백:**
기존 AI 거버넌스 연구(NIST AI RMF, EU AI Act)는 정책 프레임워크 수준에서 "무엇을 통제해야 하는가"를 정의한다. 그러나 에이전트 런타임 수준에서 "어떻게 실시간으로 통제를 집행하는가"에 대한 구현 아키텍처는 부재하다. NIST AI RMF 2026 GOVERN 매뉴얼에서 "Agentic AI Committee"와 "Agent Owner" 역할이 신설된 것은 이 공백을 인식한 증거이다.

**본 연구의 기여 가능 지점:**
Agent Hooks 파이프라인을 AI 거버넌스 정책의 런타임 집행 메커니즘으로 제안. 이는 NIST RMF의 Govern→Map→Measure→Manage 4기능 중 Manage 기능의 구체적 구현 아키텍처에 해당한다.

#### C2: HITL 워크플로 ↔ 심리적 안전 설계

CADSS의 HITL은 기술적 워크플로(승인/수정/반려)이다. AI-Human 연구의 심리적 안전은 이 워크플로를 사용하는 인간의 심리적 상태이다.

**연결 모델:**

```
HITL 효과성 = f(시스템 설계 품질, 사용자 심리적 안전)

시스템 설계 품질: 워크플로 명확성, 근거 제시 수준, 되돌림 가능성
심리적 안전: 실수 허용도, 의견 개진 자유도, 학습 기회 인식

→ 심리적 안전이 낮으면 REVIEW 상태에서 사용자가 AI 결과를
   무비판적으로 승인하는 "자동화 편향"이 발생한다.
→ 이는 CADSS의 HITL이 형식적 절차로 전락하는 실패 모드이다.
```

**검증 가능한 지표:**
- HITL 오버라이드 비율: 사용자가 AI 추천을 거부/수정한 비율
- 거부 사유 기록 비율: 거부 시 이유를 명시한 비율
- 승인 소요 시간 분포: 지나치게 빠른 승인은 자동화 편향 징후

#### C3: Skills 패키지 ↔ 절차 지식의 패키징

Agent Skills(SKILL.md)는 에이전트가 특정 작업을 수행하는 절차를 YAML 메타데이터 + 마크다운 본문으로 패키징한 것이다. CADSS의 RAG 문서는 의사결정의 근거가 되는 지식을 벡터 임베딩으로 저장한 것이다.

**비교 분석:**

| 차원 | Agent Skills | CADSS RAG |
|------|-------------|-----------|
| 저장 형식 | SKILL.md (텍스트, 버전 관리 가능) | pgvector 임베딩 (수치, 검색 최적화) |
| 검색 방법 | 파일명/디렉토리 기반 디스커버리 | 코사인 유사도 기반 벡터 검색 |
| 갱신 주기 | git commit 기반 (변경 시) | 임베딩 재생성 필요 (배치) |
| 품질 보증 | frontmatter 제약 (allowed-tools) | 근거 점수 임계값 |
| 지식 유형 | 절차적 (how-to) | 선언적 (what-is) |
| 재현성 | 높음 (동일 SKILL.md → 동일 실행) | 중간 (임베딩 모델 버전 의존) |

**통합 가능성:**
CADSS의 정책 규칙을 SKILL.md 형식으로 패키징하면, 정책의 버전 관리·배포·롤백이 git 워크플로와 통합된다. 이는 "정책 as 코드" 패러다임과 일치하며, OPA(Open Policy Agent)의 Rego 정책과 개념적으로 동일하다.

## 3. 공통 기반 기술 3개

세 축이 공통으로 필요로 하는 기반 기술을 식별한다.

### 기반 기술 1: 이벤트 기반 상태 머신 (Event-Driven State Machine)

| 축 | 적용 지점 | 상태 집합 |
|---|--------|---------|
| CADSS | 정책 엔진 분기 | {AUTO, REVIEW, APPROVAL, BLOCK} |
| Agent | 훅 파이프라인 | {Init, PreExec, Permission, PostExec, Done} |
| AI-Human | HITL 워크플로 | {대기, 검토중, 승인, 수정, 반려} |

**공통 구현체:** 단일 상태 머신 라이브러리로 세 축의 모든 워크플로를 표현할 수 있다. Python의 `transitions` 라이브러리 또는 직접 구현한 FSM이 후보이다.

### 기반 기술 2: Append-Only 감사 로그

| 축 | 기록 대상 | 무결성 보장 |
|---|--------|----------|
| CADSS | AI 요청/응답/판정/승인 전 과정 | 해시 체인 |
| Agent | 도구 실행 이력, 훅 이벤트 | 세션 로그 |
| AI-Human | 의사결정 과정, 피드백 이력 | 회의록/실험로그 |

**공통 구현체:** 모든 이벤트를 단일 append-only 저장소에 기록하고, 해시 체인으로 무결성을 보장한다. PostgreSQL의 SERIAL + SHA-256 해시가 MVP 수준에서 충분하다.

### 기반 기술 3: 불확실성 정량화 (Uncertainty Quantification)

| 축 | 불확실성 대상 | 정량화 방법 |
|---|---------|----------|
| CADSS | LLM 응답 신뢰도 | Shannon 엔트로피 + MC Dropout |
| Agent | 도구 실행 결과의 신뢰도 | 출력 검증 + 재실행 일치율 |
| AI-Human | 인간 판단의 확신도 | 자기보고 확신 점수 + 소요 시간 |

**공통 구현체:** 정규화된 불확실성 점수 [0,1]을 모든 의사결정 지점에서 산출하고, 이를 상태 머신의 전이 조건으로 사용한다.

## 4. 10회 세션 최적 배분

### 4.1 배분 근거

예비창업패키지 심사 가중치(기술성 40%, 사업성 30%, 팀역량 30%)와 현재 성숙도를 결합하여 최적 배분을 산출한다.

**계산:**
```
축별 투자 가치 = (심사 기여도) × (잔여 성장 여지) / (현재 성숙도)

CADSS:  0.40 × (50-38)/50 = 0.40 × 0.24 = 0.096
Agent:  0.15 × (50-30)/50 = 0.15 × 0.40 = 0.060
Human:  0.20 × (50-31)/50 = 0.20 × 0.38 = 0.076
통합:   0.25 × 1.0 = 0.250 (통합은 별도 가치)

정규화 후:
CADSS: 0.096/0.482 = 19.9% → 2 세션
Agent:  0.060/0.482 = 12.4% → 1 세션
Human:  0.076/0.482 = 15.8% → 1 세션
통합:   0.250/0.482 = 51.9% → 4 세션 (논문+기술서+검증)
기초:   나머지 → 2 세션 (본 세션 + 수학 정밀화)
```

|------|---|------|-----------|

## 5. 축별 최우선 해결 과제

### 축 1 (CADSS): Docker 런타임 검증 완료
